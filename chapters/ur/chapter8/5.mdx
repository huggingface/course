## ایک کم از کم دوبارہ پیدا ہونے والا نمونہ تیار کرنا[[creating-a-minimal-reproducible-example]]

یہ بہت ضروری ہے کہ اُس کوڈ کے حصے کو الگ کیا جائے جو بگ پیدا کر رہا ہے، کیونکہ Hugging Face ٹیم میں ابھی تک کوئی جادوگر نہیں ہے اور وہ اُس چیز کو ٹھیک نہیں کر سکتے جسے وہ دیکھ نہیں سکتے۔ ایک کم از کم دوبارہ پیدا ہونے والا نمونہ، جیسا کہ اس کے نام سے ظاہر ہے، دوبارہ پیدا ہونے والا ہونا چاہیے۔ اس کا مطلب ہے کہ یہ آپ کے پاس موجود کسی بیرونی فائل یا ڈیٹا پر انحصار نہیں کرے گا۔ کوشش کریں کہ جس ڈیٹا کا آپ استعمال کر رہے ہیں اسے کچھ فرضی (dummy) اقدار سے تبدیل کریں جو آپ کے اصلی ڈیٹا جیسی دکھائی دیں اور پھر بھی وہی غلطی پیدا کریں۔

<Tip>

🚨 🤗 Transformers ریپوزیٹری کے کئی مسائل حل نہیں ہو پائے ہیں کیونکہ دوبارہ پیدا کرنے کے لیے استعمال ہونے والا ڈیٹا دستیاب نہیں ہے۔

</Tip>

ایک بار جب آپ کے پاس کچھ ایسا ہو جو خود مختار ہو، تو آپ اسے اور بھی کم لائنوں کے کوڈ میں ڈھالنے کی کوشش کر سکتے ہیں، جسے ہم _minimal reproducible example_ کہتے ہیں۔ اگرچہ اس میں آپ کی طرف سے تھوڑا سا زیادہ کام درکار ہوگا، لیکن اگر آپ ایک اچھا، مختصر بگ ریپروڈیوسر فراہم کرتے ہیں تو تقریباً یقینی مدد اور حل مل جائے گا۔

اگر آپ کافی مطمئن ہیں، تو اس سورس کوڈ کا معائنہ کریں جہاں آپ کا بگ ہو رہا ہے۔ ممکن ہے کہ آپ کو اپنے مسئلے کا حل مل جائے (اس صورت میں آپ اسے ٹھیک کرنے کے لیے pull request بھی تجویز کر سکتے ہیں)؛ لیکن عمومی طور پر، یہ مینٹینرز کو آپ کی رپورٹ پڑھتے وقت سورس کوڈ کو بہتر سمجھنے میں مدد کرے گا۔

## مسئلے کے ٹیمپلیٹ کو پر کرنا[[filling-out-the-issue-template]]

جب آپ اپنا مسئلہ درج کرتے ہیں، تو آپ دیکھیں گے کہ ایک ٹیمپلیٹ موجود ہے جسے پر کرنا ہوتا ہے۔ ہم یہاں [🤗 Transformers issues](https://github.com/huggingface/transformers/issues/new/choose) کے ٹیمپلیٹ کی پیروی کریں گے، لیکن اگر آپ کسی اور ریپوزیٹری میں مسئلہ درج کرتے ہیں تو اسی قسم کی معلومات درکار ہوں گی۔ ٹیمپلیٹ کو خالی نہ چھوڑیں: اسے پر کرنے میں جو وقت صرف کریں گے، اس سے آپ کے مسئلے کا حل اور جواب حاصل کرنے کے امکانات زیادہ ہو جائیں گے۔

عمومی طور پر، جب کوئی مسئلہ درج کیا جائے تو ہمیشہ شائستہ رہیں۔ یہ ایک اوپن سورس پروجیکٹ ہے، لہذا آپ مفت سافٹ ویئر استعمال کر رہے ہیں اور کسی پر آپ کی مدد کرنے کا کوئی فرض نہیں ہے۔ آپ اپنے مسئلے میں وہ تنقید شامل کر سکتے ہیں جسے آپ جائز سمجھتے ہیں، لیکن پھر مینٹینرز اسے برا سمجھ سکتے ہیں اور جلد بازی میں آپ کی مدد کرنے کی کوشش نہ کریں۔ اس بات کو یقینی بنائیں کہ آپ نے پروجیکٹ کا [code of conduct](https://github.com/huggingface/transformers/blob/master/CODE_OF_CONDUCT.md) ضرور پڑھا ہے۔

### لوگوں کو ٹیگ کرنا[[tagging-people]]

لوگوں کو ٹیگ کرنے کے لیے ان کا GitHub ہینڈل لکھتے ہوئے `@` کا استعمال کریں، جس سے انہیں ایک نوٹیفیکیشن بھیجا جائے گا تاکہ وہ آپ کا مسئلہ دیکھ سکیں اور جلدی جواب دے سکیں۔ اس کا استعمال اعتدال سے کریں، کیونکہ جن لوگوں کو آپ ٹیگ کرتے ہیں وہ اس بات کی قدر نہیں کریں گے اگر یہ براہ راست اُن سے متعلق نہ ہو۔ اگر آپ نے اپنے بگ سے متعلق سورس فائلوں کا معائنہ کیا ہے، تو آپ کو آخری اُس شخص کو ٹیگ کرنا چاہیے جس نے اُس لائن میں تبدیلی کی ہے جسے آپ مسئلے کا ذمہ دار سمجھتے ہیں (آپ یہ معلومات GitHub پر متعلقہ لائن دیکھ کر، اسے منتخب کر کے، پھر "View git blame" پر کلک کر کے حاصل کر سکتے ہیں)۔
